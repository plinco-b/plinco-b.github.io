This pass (solo, using a throwaway testv26+throwaway1@gmail.com account so as not to disrupt Dave's live village): captured 8 more real screens — Sign in, Sign in failed, the reordered setup checklist, the "reducing sensitivity may miss distress" warning, the "Not Monitoring" safety alert, the display-asleep dark screen, the End session confirm, and — via a "Reset to fresh install" + relogin against Dave's already-3/3 village — a genuine Village full modal, richer than the old mockup (it embeds the live device list with Banish/Overthrow inline). Honesty notes for this deck
What's real vs. mockup vs. concept: every REAL screen above is an actual screenshot — most from a live iOS Simulator (iPhone 17, iOS 26.5) or Android Emulator (Pixel 9a) dev-client build of this exact codebase, and a batch from Dave's own physical iPhone (TestFlight) mid a genuine live session. MOCKUP — NOT CAPTURED screens are real, reachable states this pass simply didn't trigger (needs live cry audio, killing WiFi mid-demo, draining a real battery below 20%, etc.) — copy-verified against source. Sections ⑦–⑧ marked CONCEPT are original design work for review, not implemented.

Update since v0.25 — the two-device pairing blocker is resolved. The v0.25 pass found same-login auto-join didn't actually join the existing village; that's fixed now — confirmed live by signing into testv26@gmail.com on a fresh simulator while Dave hosted on his Pixel 3, and it joined his real village automatically. This unblocked genuine captures of every screen that used to require the MOCKUP — LIVE PAIRING BLOCKED tag: the on-call button live, mute/speaker, Safe & Free info, continuous mode, thermal override, banish, and overthrow — all real now, all captured from one genuine 3-device village (Pixel 3 host, iPhone 14 Pro chief, simulator/TestFlight villager).

Two other bugs from the v0.25 pass, status unconfirmed this round:
1. Pairing-code generation hangs. The manual invite-code field (generatePairingCode() in src/pairing/deviceLink.ts) never resolved past placeholder dashes in the v0.25 pass — not re-tested this round since same-login auto-join was the path used instead.
2. Fresh-signup FTUE skips the welcome basket. New signups route straight to baby setup, bypassing the dashboard and the PuppyDropDiscovery reveal animation — not re-tested this round (this pass used an existing test account, not a fresh signup).

New in v0.26.x, all confirmed live this pass: mic discipline (muting audio never disarms on-call), the notification "clear line" (zero permissions asked until the first deliberate "Go on call," with an honest test-alert confirm loop), the Safe & Free third control with its own info explainer, opt-in continuous monitoring with an honest overheat warning, and the thermal Override — a villager can now force video back on despite an overheating host, behind an explicit "may cause damage" confirm.

Pre-existing logging gap: src/analytics/index.ts's typed event registry covers cry/wake/answer/session-lifecycle events but has no event for sign-up/sign-in failures, the consent-checkbox-blocked tap, audio levels over time, battery level, heat/thermal events, or which network mode (Safe & Free vs. relayed) a session actually ran on — see the brand/animation/analytics catalog in presentation-assets/brand-catalog/BRAND_CATALOG.md for the full breakdown.
