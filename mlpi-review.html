Safety pass — v0.16.x (4 issues from Dave's device testing)

SHIPPED Issue 1 — Swipe-away gives no warning

Dave: "if you host a plinco session and you swipe away the app without 'ending' the session and you come back into the app, we should do the same behavior as locking."

Safety impact
Critical — baby unwatched with no notification

Root cause: On iOS, swiping the app away kills the JS process immediately, bypassing the graceful session teardown. The component cleanup (which calls stopBabyForegroundSession) never runs. On Android, the Notifee foreground service keeps JS alive so this is less severe there — but still possible via Settings → Force Stop.

Fix: Mirrored the existing onCallResume.ts pattern on the baby side.

// New file: src/baby/babySessionResume.ts // SecureStore key: 'plinco.baby.session.active.v1', TTL: 12h saveBabySessionActive(callId) // called on BabyMonitoring mount clearBabySessionActive() // called in session cleanup (graceful end) loadBabySessionActive() // checked in BabySetup on mount // If flag is set on next launch → blocking WarningModal before setup screen

WarningModal refactor: The existing WarningModal component (for the in-session lock-screen warning) was extended with optional headline, body, and cta props so the interrupted-session instance can say "This phone was NOT monitoring" / "I understand, continue" without duplicating the component.

Tests: New babySessionResume.test.ts — 3 describe blocks pinning the module interface, baby.tsx wiring, and sleepTrainer reset. babyAppState.test.ts updated for the label={cta} refactor.

SHIPPED Issue 2 — "All quiet" shown in test_always mode

Dave: "When you use the always distress mode, it shows as 'all quiet' — it shows as distress red for like 3 seconds, then it says all quiet pretty quickly."

Root cause: distressActive is only true for 3 seconds after each cry-detector pulse (DISTRESS_WINDOW_MS = 3_000). The cry detector has a 30-second cooldown. Between pulses, distressActive = false, so the two-way ternary fell through to "All quiet" — even though test mode is supposed to be always distressed.

Fix: Three-way ternary:

- {distressActive ? 'Distress detected' : 'All quiet'} + {distressActive ? 'Distress detected' : testMode ? 'Test mode — always active' : 'All quiet'}

"All quiet" is now unreachable while testMode is true. The existing TEST MODE banner (red-bordered, danger color, shows the timer) already gave the always-on signal; this fix makes the sub-label consistent with it.

Tests: New assertion in babyTestMode.test.ts pinning the three-way ternary. babySleepyMode.test.ts updated to the same pattern.

LEAD / MID Issue 3 — Parent kicked out of session when phone locks

Dave: "I was on call on a phone. And I left the app and then locked the phone that was on call. Now it says waiting for parent on the plinco session. Relaunched the app, yeah looks like I was kicked out of the session."

Safety impact
High — parent loses coverage silently

Root cause (multi-step race):

  1. Parent locks phone → OS may throttle JS → parentOnlineAt beat goes stale (>90s)
  2. Baby shows "Waiting for parent" → switches from 30s to 10s heartbeat
  3. Parent relaunches → loadOnCallResume() restores selectedSessionId
  4. Critical window: mode is briefly 'choosing' (not yet 'on-call') while the session-discovery effect runs
  5. In this window, useActiveSessions may return [] transiently (watcher remounted with fresh mountedAt, baby's only beat is pre-mount → pre-mount leftover → grace timer arms)
  6. The markedDeadAt mode-guard (v0.16.x cycling fix) only skips the write while mode === 'on-call' — but mode is still 'choosing' here → guard is inactive
  7. If grace fires before fresh beat: babyOnline = false → markedDeadAt written → session filtered out → clearOnCallResume() called → parent permanently kicked out

Proposed fix (mid-level, ~20 lines):

// src/parent/onCallResume.ts — stamp resume time when loading export async function loadOnCallResume(): Promise<OnCallResume | null> { const r = await readFromSecureStore(); if (r) r.resumedAt = Date.now(); // NEW — stamp the moment of cold resume return r; } // app/(app)/parent.tsx — session-drop effect, add resume grace const RESUME_GRACE_MS = 15_000; // 15s — enough for watcher to get a fresh beat // In the session-drop effect that calls clearOnCallResume(): if (sessions.length === 0 && selectedSessionId) { const resume = onCallResumeRef.current; if (resume?.resumedAt && Date.now() - resume.resumedAt < RESUME_GRACE_MS) { return; // still in resume grace window — don't clear yet } setSelectedSessionId(null); clearOnCallResume(); }

Alternative approach: Hoist the mode rehydration to run synchronously before the session subscription re-fires, so mode === 'on-call' is true and the existing mode guard covers the resume window. More elegant but requires careful ordering of effects in parent.tsx.

Test to write: parentOnCall.test.ts — assert the session-drop effect does NOT call clearOnCallResume() within RESUME_GRACE_MS of loadOnCallResume() resolving. Assert RESUME_GRACE_MS constant exists.

Files to touch: src/parent/onCallResume.ts, app/(app)/parent.tsx (session-drop effect, resume grace window).

SHIPPED Issue 4 — Auto-distress timer fires at wrong interval

Dave: "I had the auto-distress behavior set to 1 min but the call came at 5 min (the default). Is this a bug?"

Root cause: SleepTrainer is a module-level singleton. Previous session left it in 'waiting' or 'escalated' state. On next session start, onWakeSignal() returns early (state !== 'idle'), new session's signals are ignored, and the old evaluate() tick fires using the previous session's waitMinutes. Banner showed "1 min" but the singleton was still running the old "5 min" timer.

// src/sleepTraining/stateMachine.ts — new public method reset() { // Wipe all singleton state before the new session bridge is wired. // Called by baby.tsx at the START of each session. this.cleanup(); } // app/(app)/baby.tsx — in the [callId] configure effect sleepTrainer.reset(); // ← NEW: clear stale singleton state first sleepTrainer.configure({ ... }); // then wire the new session bridge

Why order matters: If configure() came first, a late onWakeSignal from the old session could fire through the new session's bridge, crossing session boundaries.

Tests: Third describe block in babySessionResume.test.ts — asserts reset() exists, calls cleanup(), and that baby.tsx calls sleepTrainer.reset() before sleepTrainer.configure().

Shipped: performance & copy fixes

PERF Isolated elapsed timer — BabyMonitoring re-render elimination

The biggest single performance issue I found. This is the code change shipped in this review.

Impact
High — Pixel 3 runtime

Problem: setElapsed(Date.now() - startedAtRef.current) fired every 1000ms inside BabyMonitoring, causing the entire 500+ line component to re-render. This includes:

  • All Firestore snapshot watchers (re-evaluate closures)
  • Camera state, streaming state, WebRTC refs
  • Distress state machine, flock opacity, icon rotation shared values
  • The resume-warning modal, test-mode banner, answered-banner

On a Pixel 3 (the target baby device), that's ~60 unnecessary re-renders per minute of a complex component tree during the critical "baby is sleeping" monitoring phase.

Fix: Extract the timer into a React.memo child component (ElapsedTimer). It receives startedAtRef (a stable ref), runs its own setInterval, and re-renders only itself. The AppState gap-correction in BabyMonitoring continues to work because it mutates the shared ref — the child reads it on the next tick.

// Before: inside BabyMonitoring (re-renders EVERYTHING every second) - const [elapsed, setElapsed] = useState(0); - useEffect(() => { - const t = setInterval(() => setElapsed(Date.now() - startedAtRef.current), 1000); - return () => clearInterval(t); - }, []); // After: isolated React.memo component (re-renders only the timer text) + const ElapsedTimer = React.memo(function ElapsedTimer({ + startedAtRef, + }: { + startedAtRef: React.MutableRefObject<number>; + }) { + const [elapsed, setElapsed] = useState(0); + useEffect(() => { + const t = setInterval(() => setElapsed(Date.now() - startedAtRef.current), 1000); + return () => clearInterval(t); + }, [startedAtRef]); + return <Text style={timerStyle}>{formatElapsed(elapsed)}</Text>; + });

Test updated: babyAppState.test.ts now asserts the ref ownership is in BabyMonitoring and the tick interval exists in the file (not scoped to the monitoring function body). All 43 suites / 525 tests green.

Finding 1: Duplicated sleepy-mode state machine

The baby.tsx and parent.tsx on-call screens independently implement the same sleepy-mode animation state machine:

  • Both track flockVisible / flockMounted with identical timeout logic
  • Both implement the pendulum rock animation with the same iconRotation shared value pattern
  • Both have flockOpacity with identical fade-in/fade-out durations (900ms / 600ms)
  • Parent.tsx even has a comment "Same split as baby.tsx"

This is ~120 lines of duplicated animation state. If the architect wants to change the sleepy-mode timing (e.g., shorter fade-in for a more responsive feel), they'd have to change it in both files and hope the tests catch a drift.

Automation proposal

Pre-commit: baby-screen animation guard

The "old hardware first" principle has a specific crash history: Reanimated worklets on BabySetup/BabyMonitoring crashed Pixel 3 in v0.10.11. The current prevention is humans reading DEV_CONTEXT. That's fragile.

Proposed hook addition (pre-commit)

# In the existing pre-commit hook, after tsc + jest: BABY_DANGER_PATTERNS="useAnimatedScrollHandler|transformOrigin|runOnUI|scrollTo" if git diff --cached --name-only | grep -q 'baby\.tsx'; then MONITORING=$(sed -n '/function BabyMonitoring/,/^function /p' app/\(app\)/baby.tsx) if echo "$MONITORING" | grep -qE "$BABY_DANGER_PATTERNS"; then echo "⚠️ baby.tsx BabyMonitoring contains animation patterns that crashed Pixel 3." echo " Patterns: $BABY_DANGER_PATTERNS" echo " See DEV_CONTEXT.md § 'Old hardware first'" exit 1 fi fi

This catches the class of crash that killed v0.10.11 at commit time, not at device-test time. The current PulseCircle + icon-rock animations are safe (they use useSharedValue + withTiming, not scroll-driven transforms).
